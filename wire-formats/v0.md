# ORS Wire Format v0: Single OP_RETURN

## Document Status

**Version:** 0.1 (Draft)
**Status:** MVP Specification

## Abstract

Version 0 of the ORS wire format. Embeds the full payload in a single OP_RETURN output. Minimum payload size is 102 bytes.

## Protocol Format

### Base Structure

```
OP_RETURN <magic> <version=0x00> <pubkey> <sig> <kind> <kind_data...>
```

| Field       | Size     | Value                 | Description                                                 |
| ----------- | -------- | --------------------- | ----------------------------------------------------------- |
| `magic`     | 3 bytes  | `0x4f5253` ("ORS")    | Protocol identifier                                         |
| `version`   | 1 byte   | `0x00`                | Protocol version v0                                         |
| `pubkey`    | 32 bytes | x-only Schnorr pubkey | Author's public key (BIP-340)                               |
| `sig`       | 64 bytes | Schnorr signature     | Signature over sha256(unsigned_payload) - see Signing below |
| `kind`      | 1 byte   | see kind registry     | Content kind (required, at offset 100)                      |
| `kind_data` | variable | see kind spec         | Kind-specific data, starts at byte offset 101               |

**The header is 101 bytes fixed. The kind byte is at offset 100. Kind-specific data begins at byte offset 101.**

### Signing

The signature covers the entire OP_RETURN payload, excluding the sig field itself:

```
unsigned_payload = magic || version || pubkey || kind || kind_data
message          = sha256(unsigned_payload)
sig              = schnorr_sign(privkey, message)
final_payload    = magic || version || pubkey || sig || kind || kind_data
```

This ensures the signature binds the author's pubkey, the content kind, and the content - not just the content string alone. The magic and version bytes are included so the signature is specific to this protocol.

## Transaction Requirements

### Valid ORS v0 Post

1. **Exactly one OP_RETURN output** with valid ORS data
2. **Magic bytes** must be `0x4f5253`
3. **Version** must be `0x00`
4. **`pubkey` field** must be present (32 bytes at offset 4)
5. **`sig` field** must be present (64 bytes at offset 36)
6. **`kind` field** must be present (1 byte at offset 100)
7. **Signature** must be a valid Schnorr signature (BIP-340) of sha256(unsigned_payload) verified against pubkey
8. **Minimum payload length:** 102 bytes (101-byte header + 1 byte minimum kind data)
9. **Transaction must be mined** (unconfirmed ignored for MVP)

### Invalid Posts (discarded by parsers)

- Wrong magic bytes
- Version byte is not `0x00`
- Payload shorter than 102 bytes
- Missing or malformed pubkey/sig/kind fields
- Invalid Schnorr signature
- Unknown kind (skip gracefully)

## Discovery

Clients scan for OP_RETURN outputs starting with `0x4f5253 0x00` (ASCII "ORS" followed by version byte 0x00).

See [../ORS-01.md](../ORS-01.md) for the base protocol spec. Kind specs are in the [ORK repo](https://github.com/opreturnsocial/ork).
