# ORS Wire Format v1: Chunked 80-byte

## Document Status

**Version:** 0.1 (Draft)
**Status:** MVP Specification

## Abstract

Version 1 of the ORS wire format. Splits the payload across multiple OP_RETURN outputs, each capped at 80 bytes. This fits within the 80-byte `datacarriersize` enforced by the Bitcoin Core default relay policy, making v1 posts minable on mainnet without special miner configuration.

The tradeoff is cost: a v1 post requires multiple transactions (one per chunk), each paying its own on-chain fee.

See [v0.md](v0.md) for the single-output v0 format.

---

## Wire Format

### Signing Body (v1)

The signing body for v1 deliberately excludes the magic and version prefix to save 4 bytes across the chunk budget:

```
signing_body = pubkey(32) || kind(1) || kind_data
message      = sha256(signing_body)
sig          = schnorr_sign(privkey, message)
```

The pubkey is included in the signing body so the signature is author-bound without relying on a magic header.

### Body (split across chunks)

```
body = pubkey(32) || sig(64) || kind(1) || kind_data   (97+ bytes)
```

The body is the wire data distributed across chunks. It is NOT the signing body - it includes the sig.

### Chunk Format

**Chunk 0 (root):**
```
ORS(3) | 0x01(1) | 0x00(1) | total_chunks(1) | body[0:74]
 = 6-byte header + up to 74 bytes of body = 80 bytes max
```

| Field          | Size    | Description                                      |
|----------------|---------|--------------------------------------------------|
| `magic`        | 3 bytes | `0x4f5253` ("ORS")                               |
| `version`      | 1 byte  | `0x01`                                           |
| `chunk_num`    | 1 byte  | `0x00` (always zero for root chunk)              |
| `total_chunks` | 1 byte  | Total number of chunks (including this one)      |
| `body_slice`   | up to 74 bytes | First 74 bytes of body                  |

**Chunk N (N >= 1):**
```
ORS(3) | 0x01(1) | N(1) | body[74 + (N-1)*75 : 74 + N*75]
 = 5-byte header + up to 75 bytes of body = 80 bytes max
```

| Field          | Size    | Description                                      |
|----------------|---------|--------------------------------------------------|
| `magic`        | 3 bytes | `0x4f5253` ("ORS")                               |
| `version`      | 1 byte  | `0x01`                                           |
| `chunk_num`    | 1 byte  | Chunk index (1, 2, 3, ...)                       |
| `body_slice`   | up to 75 bytes | Body slice for this chunk                |

Non-root chunks do NOT contain `total_chunks`. The assembler derives the total from chunk 0.

**The canonical post id is the txid of the chunk 0 transaction.**

---

## Chunk Count Formula

```
bodyLen = 97 + len(kind_data)   // pubkey(32) + sig(64) + kind(1) + kind_data
numChunks = 1 + ceil(max(0, bodyLen - 74) / 75)
```

### Capacity Table

| Chunks | Body bytes | Max kind_data | ASCII chars |
|--------|-----------|--------------|-------------|
| 2      | 149       | 52           | 52          |
| 3      | 224       | 127          | 127         |
| 4      | 299       | 202          | 202         |
| 5      | 374       | 277          | ~277        |

5 chunks covers the 280-byte MAX_CONTENT_BYTES application limit.

## Assembly Algorithm

Clients (cache-servers, wallets) assemble v1 posts by:

1. Scan OP_RETURN outputs for ORS magic + version byte `0x01`
2. Parse chunk header (`chunk_num`, `total_chunks` if chunk_num==0, `body_slice`)
3. Store `PendingChunk` records keyed by `txid`
4. After each block, for each chunk 0 in the window (last 6 blocks):
   - Collect candidates for each chunk index 1..total_chunks-1
   - Try all combinations (brute-force cartesian product)
   - For each combination: concatenate body slices, parse assembled body
   - Verify Schnorr signature: `sha256(pubkey || kind || kind_data)` vs `sig`
   - If valid: store post with txid = chunk0.txid, delete assembled PendingChunks
5. Discard PendingChunks older than 6 blocks

**Note:** Brute-force sig verification is used instead of embedding a root txid in non-root chunks to avoid wasting block space on a 32-byte reference.

## Transaction Requirements

### Valid ORS v1 Post

Each chunk is a separate bitcoin transaction:

1. **Chunk 0 transaction** with OP_RETURN output <= 80 bytes
2. **Chunk N transactions** (N >= 1) with OP_RETURN output <= 80 bytes
3. **All chunks mined** (any chunk missing = post not assembled)
4. **Signature valid:** `verifySchnorr(sha256(pubkey || kind || kind_data), pubkey, sig) == true`
5. **total_chunks in chunk 0** must match the number of chunks actually broadcast

### Discovery

Clients scan for OP_RETURN outputs starting with `0x4f5253 0x01` (ASCII "ORS" followed by version byte `0x01`).

See [../ORS-01.md](../ORS-01.md) for the base protocol spec. Kind specs are in the [ORK repo](https://github.com/opreturnsocial/ork).
